## 一、定位（一切决策的起点）

```
账户：20 万小账户（A 股）
策略：趋势跟随（买异动动量延续，不抄底不猜顶）
风险偏好：可接受大回撤（换在场率）
目标持仓：3-5 只集中（非分散）
核心矛盾：小账户 × 高价股 × 趋势要在场
```

这五个定位决定了后面所有参数。如果定位变了（比如账户变大到 100 万，或转向价值投资），大部分参数需要重调。

## 二、架构总览

### 数据流（scan → rank → rebalance → backtest）

```
雪球异动榜
   │ scan_universe.py（5207 股全量）
   ▼
snapshot.py（逐只查异动，~10min，并发 6）
   │
   ▼  raw/snapshot-<date>.json
diff-cli（窗口期异动变化）
   │
   ▼  derived/diff-<date>.json
candidates-cli（ST/退市/科创板过滤）
   │
   ▼  derived/candidates-<date>.json  ← 候选池（无价格）
rank-cli（LLM 精排 top-N）
   │
   ▼  scan/<date>/scan.json          ← top_picks（无价格）
rebalancer（shallow-analyzer 拉数据 + LLM 出 action）
   │   ├─ position-calculator：fitness×0.022×波动率 → 目标仓位
   │   ├─ constraint-validator：单仓/行业/现金/anti-churn 七规则
   │   └─ revise 循环：违反约束→LLM 重出（最多 2 次）
   ▼  rebalance/<date>/plan.json     ← 含 industry（申万二级）
backtest-simulator（连续持仓模拟）
   │   ├─ normalizeWeights：价格漂移重算权重
   │   ├─ applyPlan：BUY/SELL/ADD/REDUCE + 手数取整
   │   └─ recordNav：每日 NAV 快照
   ▼  backtest/state.json            ← 增量续跑状态（只管持仓，不含决策原因）
   ▼  backtest/days/<date>/rebalance.json  ← 每日完整调仓决策归档
       （evaluations + actions[reason] + summary + reports[fitness] + 约束博弈）
backtest-cli --report
   ▼  backtest/run-<ts>/report.md    ← 含评分校准段
backtest-cli --day <date>
   ▼  终端渲染归档决策（人类可读视图）
```

### 四层仓位计算

```
LLM 出方向（BUY/SELL/HOLD/...）
        │ 不出数字，只出判断
        ▼
① 单股目标仓位 = fitness × 0.022 × 波动率折扣，single_name 截断
        │ computePosition（position-calculator.ts）
        ▼
② 全局现金排队：BUY 按 fitness 降序累计，超 spendable 的降级 HOLD
        │ applyPositions（position-calculator.ts）
        ▼
③ 硬约束校验：7 条规则，违反→revise
        │ constraint-validator.ts
        ▼
④ 成交手数取整：真实股数取整到 100 股，不足 1 手跳过
        │ backtest-simulator.ts applyPlan
```

## 三、演进时间线（10 个阶段）

### 阶段 0：起点——新闻数据源问题（06-26）

**问题**：中科曙光 data-trace.html 里新闻日期错乱（2024 年的半年报新闻混入），pywencai/东财返回的是"相关性排序"非"时间排序"。

**决策**：新闻源重构——pywencai 主源 + 东财 search 兜底，窗口调到 30 天 + 相关性过滤挡纯板块新闻。提交 `af57599`。

**教训**：数据源的排序语义必须核实，不能假设 API 默认按时间。

---

### 阶段 1：rebalance 定位之争（06-27 上午）

**问题**：异动股池本质是"涨幅异动筛选"，A 股以炒作为主，备选常被 deal_breaker 跳过。用户质疑 rebalance 的价值。

**关键讨论**：用户明确——"我想让它自主决策，尽量不介入，可接受大回撤，用小账户做策略"。

**决策**：定调为**趋势跟随自主策略**。rebalance 的价值不是"帮用户决策"，而是"自主完成 scan→rank→深度分析→仓位→约束的全链路"。这决定了后续所有设计服务于"自主 + 趋势"。

---

### 阶段 2：趋势模式重构（06-27，commit `1bc2d68`）

**问题**：原 rebalancer 偏价值投资（PE/估值否决），和趋势定位冲突。

**决策**：
- 驱动逻辑评分（fitness 0-10）替代估值否决
- 技术位止损（支撑/压力/MACD）替代"高估就卖"
- 线性仓位（fitness×系数）替代"低估重仓高估轻仓"
- risk=high 不再压缩仓位（靠止损退出，不靠仓位）

**参数初值**：baseWeight 系数 0.008，single_name 0.10，cash_reserve 0.05。

---

### 阶段 3：增量回测模块（06-27，commit `20fcd5f` + `116e3cb`）

**问题**：单次全量回测要 20 分钟，用户"以为卡住了"。需要看到每天进度。

**决策**：
- `PositionSimulator.serialize()`/`fromSerialized()`：持仓状态可序列化
- state.json 跨进程持久化（STATE_VERSION 版本控制）
- 增量模式：每次只跑下一天，跑完存盘，断了能续
- 三种入口：默认增量 / `--date` 单日 / `--dates` 慢路径全量

**教训**：长任务必须有进度可见性（`[data] N/total` + `[llm] N/total` 日志），否则用户体验灾难。

---

### 阶段 4：仓位系数初调（06-27，commit `6cc501c`）

**问题**：0.008 系数下 fit8 才 6.4%，对 20 万账户 5-8 只目标仓位太轻。

**决策**：用户选"适度集中"，系数 0.008→0.015，single_name 0.10→0.15。

**当时认知**：以为是参数太小，调大即可。后来阶段 8 发现真正的根因是 fitness 评分标准偏严（见决策推理链 D）。

---

### 阶段 5：三层止损系统（06-27，3 个 commit）

**问题**：回测发现 anti-churn 锁（持仓<7天禁卖）挡住了所有止损信号，导致 0 次止损触发。

**决策**：三层止损，逐层解决不同的"卖不出去"问题：

| 层 | 解决的问题 | commit |
|----|-----------|--------|
| ① stopLossSignal 豁 anti-churn 锁 | risk=high/高危flag 被 7 天锁挡住 | `68b9b9b` |
| ② 建仓回撤止损 | 国瓷 -8.3% 没触发（建仓3天内） | `6e95edc` |
| ③ 技术破位强制 REDUCE | 生益 risk=high 但 PM 出 HOLD | `29bed07` |

**关键设计**：`held` map 的 `stopLossSignal` 字段在 position-calculator 之前**预判**，因为 validator 检查 anti-churn 锁在 position-calculator 之前。预判逻辑：overall_risk=high OR severity=高 OR deal_breaker OR 建仓回撤超阈值。

**第三层的特殊性**：代码强制减半（不留给 PM 犹豫）。因为发现 LLM 在 risk=high + 技术破位时仍出 HOLD，靠 LLM 自主止损不可靠。

### 阶段 6：手数取整（06-28，commit `4f00e7c`）

**问题**：回测 `shares = targetValue/price` 不取整，245 元香农 8% 仓位算出 65 股，A 股最小 1 手=100 股实际买不了，回测却假装买进 → NAV 失真。

**决策**：见决策推理链 A。backtest-simulator 加 `realCapital` + `lotSize` 配置，BUY/ADD/REDUCE 成交股数取整，不足 1 手跳过并记录 `skippedBuys`。

**验证**：7 天回测跳过 6 只高价股（香农245/生益184/国瓷99/大族346/兆易775/法拉190），全是电子产业链。

### 阶段 8：方向B 集中化（06-28，commit `4284527`）

**问题**：7 天回测仓位太轻（7 天才 38%，前 5 天几乎空仓），资金严重闲置。

**诊断**：见决策推理链 C/D。死循环：fitness 偏低（avg 5.3, max 8）→ baseWeight 给小仓 → 3-4 只集中也满不了仓（42% 封顶）。

**决策**：方向B-3（参数+评分双调）：
- baseWeight 0.015→0.022
- single_name 0.15→0.22
- cash_reserve 0.05→0.03
- 持仓目标 8-12→3-5
- fitness 评分标准放宽（7-8 分门槛从"已落地订单"降到"趋势正在验证"）

**验证**：新旧版对比见第六节。仓位 38%→63%，收益 +2.3%→+5.3%。

### 阶段 9：评分校准（06-28，commit `6470df9`）

**问题**：`fitness × 0.022` 仓位公式的隐含假设"高分股涨得好"从未验证。如果评分失准，整个仓位逻辑的地基就是歪的。

**决策**：见决策推理链 D。Position 加 `entry_fitness`，TradeRecord 加 `entryFitness`，回测报告新增"评分校准段"，按高/中/低分档统计平均收益，自动判断校准质量。

**当前状态**：基础设施就位，但 7 天样本不足（需 30+ 笔交易），真正的校准结论待数据积累。

### 阶段 10：调仓决策归档（06-28，可审计性修复）

**问题**：回测只能看到"结果"（NAV 曲线 + actions 简表），看不到"过程"——某天为什么买/卖某只股、跳过哪些候选、约束博弈了几轮，全部丢失。原因：`runSingleDay` 拿到了完整的 `rebalancer_output`（含 evaluations / 每个 action 的 reason / summary / reports），却只取 actions 的方向和名字，把 reason / summary / evaluations 全扔了。详细决策只残留在 `os.tmpdir()` 的 LLM trace 里，Windows 重启即清空。

**决策**：见决策推理链 E。每天把完整 `rebalancer_output` + `portfolio_before` + 每股 fitness/risk/thesis 报告 + 约束 revise 过程，原样归档到 `backtest/days/<date>/rebalance.json`。加 `--day <DATE>` 命令渲染成人类可读视图。

**职责分离**：`state.json` 只管持仓模拟（精简，已是增量续跑的状态机）；调仓"为什么"归 `days/<date>/`（按日独立文件，可审计、可 diff、可随时翻看）。两者不耦合——某天的决策归档坏了不影响持仓模拟续跑。

**当前状态**：已落地。历史 7 天需 `--reset` 重跑才能补归档（归档只在跑当天生成，无法事后回填，因为 LLM 输出是即时的）。

### 阶段 11：死亡螺旋修复——持仓数硬约束 + 换手率单向算法（06-28）

**问题**：归档功能上线后，重跑 7 天回测暴露致命缺陷——最后 3 天（06-24/25/26）连续 `constraint_violation`，策略卡死无法调仓。用户敏锐发现"06-26 持仓 7 只，我们不是默认 5 个持仓吗"。

**诊断（两层叠加）**：
- **根因（上游）**："3-5 只"只是 prompt 软引导，validator 11 条规则里**没有持仓数约束**。LLM 无视它从 5 只买到 7-8 只（06-23 是失控点：已 5 只还又买 3 只）。
- **并发症（下游）**：满仓后想"换仓减员"（卖 3 买 2），换手率 = 双向 sum(|delta|) = 33.8%+21.1% = 54.9%，必然超 40% 上限。**数学上无解**——revise 多少次都没用。
- **死亡螺旋**：持仓失控 → 满仓 → 想减员 → 换手率超限 → 被否决 → 持仓降不下来 → 永远卡死。

**决策**：见决策推理链 F（三层修复）。

**当前状态**：已落地。max_positions=5 硬约束 + 换手率改单向 max(买,卖) + daily_turnover 0.40→0.50 + max_revise_retries 2→3 + revise 反馈加可执行指引。

---

### 阶段 12：撞锁死磕修复——止盈豁免 + 撞锁变通反馈（06-28）

**问题**：阶段 11 修好换手率死亡螺旋后，重跑回测暴露下一层问题——卡死主因变成了 **anti-churn 卖锁撞锁死磕**。06-23/26 LLM 想卖豫光金铅（持仓 1-4 天，locked），revise 3 次都死磕同一只，不会变通，整天调不动仓。

**用户质疑**："7 天是合理的吗？还是 3 天或者 5 天"——这个质疑推动了深入分析。

**诊断（用数据回答"7 天合不合理"）**：
- `anti_churn_days: 7` 用的是**自然日**（含周末），实际锁 ≈5 个交易日，比直觉的"一周"还重
- **豫光金铅 case 证明锁的价值**：06-22 建仓，06-23（持仓 1 天）LLM 想卖，被锁；到期末 +7.6%。如果当天卖了就错过这 7.6%——锁对了
- **所以 7 天不该改**（数据支持），真正的缺陷是两个：①撞锁后 LLM 不会变通 ②锁一刀切，不区分止损/止盈

**决策**：见决策推理链 G（两层修复，不动 7 天）。

**当前状态**：已落地。止盈豁免（take_profit_threshold=0.15）+ 撞锁 revise 反馈（教会变通）。anti_churn_days 保持 7。

---

### 阶段 13：行业偏科卡死修复——revise 反馈给非超限行业强标的清单（06-28）

**问题**：阶段 12 修好撞锁后，最后两天（06-25/26）仍 constraint_violation，但卡死原因变成了**单行业上限**（电子 37% > 25%）。

**诊断（深挖到候选池偏科）**：
- 表面：LLM 想买电子股，一买就踩 25% 上限
- 深层：候选池 top_picks 20 只里**电子占 60%+**，fitness≥7 的强标的中电子占 5/7
- 根源：当时市场主线是半导体周期复苏，scan 趋势扫描自然选出大量电子股（合理反映市场结构）
- 关键事实：候选池**有非电子强标的**（中科曙光 fit8、巨化股份 fit8、中国巨石 fit7），LLM 只是没优先选它们

**这是 scan 的 bug 吗？不是。** 趋势策略就是会集中到最强主线，候选池偏科是市场结构的真实反映。问题在 rebalancer 处理太笨——撞电子上限还继续往电子里挤（一边卖电子一边买电子），没"跨行业选股"意识。

**决策**：见决策推理链 H。改 revise 反馈，撞行业上限时给 LLM 具体的非超限行业强标的清单（从 reports 筛 fit≥7 + sector 不超限），让它有明确的转向目标。

**当前状态**：已落地。composeReviseFeedback 加 FeedbackContext（overSectors + reports），行业超限时反馈"电子已满，可转向配置：中科曙光(计算机设备,fit8)、巨化股份(化工,fit8)..."。

**深挖后发现单层不够，补成三层**：
- 只改转向反馈，但用旧 scan 重跑验证时发现：非电子强标的**根本没进候选池**（被共振压在 top11-14），转向反馈无料可用
- **rank 去「板块共振加分」**：删 LONG/SHORT prompt 的共振鼓励（周围都是电子就加分），从源头减少扎堆。验证效果显著——巨化股份(化工)从 top12 升到 top3，彤程新材、锐科激光等非电子中坚也进 top10
- **top_n 10 → 15**：scan top_picks 一般 14-15 只，top_n=10 裁掉了非电子中坚。扩到 15 = 持仓数(5)的 3 倍，给跨行业选股留空间

**验证结果（新 scan + top15 + 转向反馈，全量重跑 7 天）**：
- **行业违规：2 天 → 0 天** ✅（电子不再超 25%）
- 持仓跨 **4 个不同行业**（中科曙光计算机 / TCL电子 / 广信材料化工 / 株冶集团有色）
- 撞锁/换手卡死基本消失
- 总收益 +1.4%，回撤 -0.8%（更分散更健康，但收益偏低——分散的代价）
- ⚠️ 收益对比阶段 12 的 +4.3% 下降，主因：①更分散（之前高收益靠失控的电子偏科）②换仓更频繁（4 笔平仓，摩擦高）③生益科技 fit9 亏 -6.8% 拖累

### 阶段 14：手续费建模——成交扣佣金+印花税+过户费（06-28，commit `9af403d`）

**问题**：回测按收盘价全额成交，零成本，收益虚高。趋势模式换手频繁（7 天 28 笔），成本影响被放大。

**方案**：backtest-simulator 成交时按真实成交额扣成本，仅影响 NAV，不进 LLM 决策层。费率定义在 `DEFAULT_TRADING_FEES`：
- 佣金万一（0.0001，最低 5 元/笔），买卖双向——用户实盘费率
- 印花税万五（0.00005），仅卖出——2023.8.28 降税后
- 过户费万零点一（0.00001），买卖双向——沪深统一

**影响**（7 天 28 笔，本金 20 万）：总成本 ≈150 元，占利润 5.4%，收益 +1.40% → ≈+1.32%。28 笔单笔金额均 <5 万，全部触发最低 5 元佣金。单笔已平仓成本记录在 `TradeRecord.cost`（10-11 元/笔，可审计）。

**设计要点**：成本按 lotRounding 后实际成交额算（非 target_weight）；未启用 lotRounding 时不扣（向后兼容）；滑点未建模（future work）。详见 `docs/backtest-params.md` 交易成本段。

### 阶段 15：默认模型 glm-5-turbo → glm-5.2 + evaluations/actions 语义分离（06-29）

**起因**：06-26 空仓重跑 rebalance 时，前台命令 3 次撞 10 分钟超时。trace 分析发现 glm-5-turbo 单次 LLM 调用 8-226s（巨化股份一次 226s 阻塞整个并发槽），15 只串行（并发=2）总耗时 15 分钟。诊断到模型层面后做了 A/B 对比。

**A/B 对比（同数据 15 候选空仓，trace 级量化）**：

| 维度 | glm-5-turbo | glm-5.2 |
|------|-------------|---------|
| 总耗时 | 898s（15min） | 335s（5.6min），**快 2.6×** |
| 平均/次 | 29.0s | 11.2s |
| 最慢单次 | 226s（巨化） | 41s（新安） |
| fitness avg | 5.4 | 6.1（修正 turbo 系统性压分） |
| revise | 1 次 | 0 次 |
| 成本 | $0.0340 | $0.0346（持平） |

**turbo 的系统性压分**：火炬电子（+147% 持续趋势的 MLCC 龙头）turbo 只给 4 分，glm-5.2 给 7 分；国泰集团 6→9。turbo 倾向低估"趋势已验证+逻辑清晰"的标的。

**选股完全分叉的真相**：两模型买入 0 重合，深挖发现不是"审美分歧"：
- 6+ 只 fit≥7 的候选里选 3 只，数学上撞 0 个的概率不低（正常）
- 但 turbo 对被砍标的的 evaluation brief 写的是**"额度已满/换手超限/下次再评估"**——这是 revise 过程的约束理由污染了真实评估（异常）

**污染的根因**：turbo 首轮输出 7 个 BUY → 撞换手/持仓上限 → validator 打回 → revise 重写时，把约束反馈抄进了 evaluations 的 brief，导致归档理由失真（看似"不看好"，实则"被约束否决"）。

**修复（方案 A：prompt 层分离）**：
1. rebalancer prompt 输出格式段新增「evaluations 与 actions 语义分离」——evaluations 是基于逻辑/趋势/风险的真实评估，与换手/持仓/行业约束无关；actions 才受当日约束。强标的因约束买不进，evaluation 仍应是 BUY
2. composeReviseFeedback 末尾强调：只调 actions，不要把约束理由写进 brief

**稳定性验证（glm-5.2 + 新 prompt，06-26 跑两遍）**：

| 维度 | run1（旧prompt） | run2（新prompt） |
|------|------------------|------------------|
| 选股 | 国泰/润禾/晶瑞 | 国泰/润禾/晶瑞 + 康强（**3 核心标的 100% 一致**） |
| fitness avg | 6.1 | 6.2（6/15 完全一致，9 只有 ±1 波动，核心标的稳定 8） |
| 总耗时 | 349s | 370s（波动 <6%） |
| revise | 0 | 1（但 evaluations **零污染**） |

**结论**：glm-5.2 + 新 prompt 组合可靠。选股方向稳定，fitness 波动在可接受区间（核心买入标的稳定），耗时无长尾，约束污染彻底消除。

**落地**：rebalance-cli/backtest-cli 默认模型改 glm-5.2；index.ts 错误提示里过时的 glm-5-turbo 改 glm-4-flash（plugin pipeline 默认本就是 flash）。commit `10c9a0e`（模型切换）+ `bb20768`（prompt 分离）。

## 四、关键决策的推理链

### 决策 A：手数取整为什么在成交层，不在候选池过滤？

**用户原问**："先解决高价股问题，能直接候选池过滤吗"

**诊断**：数据流是 `candidates.json（无价格）→ rank → scan.json（无价格）→ rebalancer（拉 K 线才有 last_close）`。`filterCommon`（ranker.ts）拿到的 CandidateEntry **没有 price 字段**。价格要到 rebalancer 阶段才出现。

**结论**：候选池过滤做不了——在没价格的地方过滤价格，逻辑上不成立。

**更深一层**：固定价格阈值（如>200 元过滤）很粗糙。"买不买得起一手"取决于仓位大小，而仓位取决于 fitness（动态的）。245 元香农满仓 15% 能买 1 手，但 fitness 给 5% 仓位就买不起——这个判断不该在候选池做，该在成交时做。

**最终方案**：成交层手数取整。真实股数 = shares_norm × realCapital，取整到 lotSize，不足 1 手跳过。自适应仓位大小，不依赖固定价格阈值。

### 决策 B：行业分类为什么选申万一级映射，不选概念板块？

**用户原问**："A是映射成一级分类，按一级分类限制是吗？C是类似于概念板块？"

**三个候选**：
- A：申万二级→一级映射表（TS 硬编码）
- B：数据源改申万一级（Python 多一次网络调用）
- C：自定义产业链（手工定义"电子材料"等）

**为什么选 A**：

4. 做完 A 重跑回测，如果规则 3 开始触发 revise，就证明分散约束确实在起作用——那时再评估要不要上 C。

- `PCB` → 电子（非标三级标签，东财 datacenter 特有，不在申万二级表）
- 东财 BOARD_NAME 会省略Ⅱ后缀（`电子化学品` 而非 `电子化学品Ⅱ`）

**设计要点**：映射只在规则 3 聚合时做，LLM prompt 里仍展示原始二级细分（不丢信息）。映射函数幂等（`mapIndustryToL1(mapIndustryToL1(x)) === x`），可重复调用安全。

### 决策 C：方向B 为什么是 baseWeight 0.022？

**死循环诊断**：

```
fitness 偏低（avg 5.3, max 8）
   ↓ 原因：评分标准过严（"有具体数据支撑才给7分以上"门槛太高）
baseWeight 给小仓（fit8 才 12%）
   ↓ 0.015 系数
3-4 只集中也满不了仓（42% 封顶）
   ↓ 数学：3×fit8成长股×0.8波动率 = 3×9.6% = 28.8%
要么放弃集中（买更多只→又回到分散问题）
要么放弃满仓（接受低仓位→资金闲置）
```

**为什么不是只调参数（B-1）或只改评分（B-2）**：
- 只调参数：fit5 的弱驱动股也给 15%，可能买入质量不高的标的。
- 只改评分：影响面大（所有分析师输出都变），且 fitness 偏低可能是这批候选股的特殊情况。
- 两者都做（B-3）：参数让中等分能上场，评分让高分股更多。最平衡。

**0.022 的数学验证**：
- fit8 成长股（0.8 折扣）：8×0.022×0.8 = 14.1%
- fit7 大盘股（1.0 折扣）：7×0.022×1.0 = 15.4%
- 5 只 fit7：77%（接近满仓）
- single_name 0.22：让 fit10（10×0.022=22%）不被截断

### 决策 D：评分校准为什么是基础设施优先？

**核心洞察**：`fitness × 0.022` 的整个仓位逻辑，建立在一个**从未验证的假设**上——"LLM 给的分数和股票实际涨跌正相关"。如果这个假设不成立：
- 高分股不比低分股涨得好 → 重仓高分股没有 alpha
- 甚至倒挂（高分股跌更多）→ 越重仓越亏

这是比"参数调多少"更根本的问题。

**为什么现在只做记录、不做结论**：
- 校准需要"建仓时 fitness + 平仓后收益"的配对样本
- 统计显著性要求 30+ 笔交易
- 7 天回测只有 1-2 笔平仓，远不够
- 所以先铺基础设施（entry_fitness 记录 + 校准段渲染），等数据够了自动出结论

**校准段的三种结论**：
- ✓ 校准良好：高分档收益 > 低分档，单调递减 → 评分可信，仓位公式有底气
- ⚠ 校准失准：高分档 < 低分档（倒挂）→ 评分无预测力，需调 shallow-analyzer prompt
- △ 校准模糊：差异小 → 区分度不足，或样本不够

### 决策 E：调仓决策归档为什么按日独立文件，不塞进 state.json？

**用户原问**："我看你找了很久，是不是现在的代码设计不合理？"

**诊断（用户一针见血）**：调仓的 reason / evaluations / summary 只在跑回测那一刻由 LLM 生成，然后**用完即弃**。`runSingleDay` 拿到完整 `rebalancer_output` 却只取 actions 的方向和名字，其余全丢。要回溯某天为什么这么调，得去 `os.tmpdir()` 的 LLM trace 里"考古"——那是系统临时目录，重启即清空。回测因此变成黑盒：只告诉你赚了多少，不告诉你每天怎么决定的。

**为什么不塞进 state.json**：
- state.json 是持仓模拟的状态机（cash + positions + navHistory + trades），职责单一
- 调仓决策体量大（每股含 thesis/key_signals/risk_flags + evaluations + 约束 violations），塞进去会让 state 膨胀到难以人工检查
- state 每天整体覆盖写（增量续跑），决策若是其中一部分，版本/回滚会混乱
- 按日独立文件可单独打开、单独 diff 两天差异、坏了不影响持仓续跑

**为什么用 `days/<date>/` 而非 `run-<ts>/`**：
- `run-<ts>/` 是 `--report` 的产物（一次性报告，每次生成新目录）
- `days/<date>/` 与回测日一一对应，稳定路径，"想看 06-23 怎么调的"直接打开固定文件
- 目录形式（非单文件）便于后续扩展（如加每日 K 线快照、shallow-analyzer 原始报告）

**为什么无法事后回填**：LLM 输出是即时的（每次跑带随机性 + 最新数据），历史日的决策无法重现。所以归档**必须在跑当天生成**，历史日要补只能 `--reset` 重跑。这本身也是个信号：可审计性要从第一天就做，不能事后补。

**最终方案**：
- `archiveDayRebalance()`：`runSingleDay` 里 rebalance 跑完立即归档，非 ok 日也存（失败日更要看）
- `--day <DATE>`：渲染归档成人类可读视图（评估表 + 动作理由 + 组合总结 + 约束博弈）
- 测试锁定渲染契约（backtest-archive.test.ts，5 个用例）

### 决策 F：死亡螺旋怎么修——持仓数硬约束 + 换手率单向算法

**用户原问**："06-26 持仓 7 只，我们不是默认 5 个持仓吗"

**用户一针见血**：表面症状是"换手率卡死"，但用户的问题指向了更上游的根因——持仓数失控。如果只调换手率参数而不限持仓数，8 只、10 只时问题会以更严重的形式重现。

**为什么持仓数是根因（先治本）**：
- "3-5 只"之前只是 rebalancer prompt 第 17 行的一句话软引导
- validator 11 条规则（权重和/单仓/单行业/换手/现金/anti-churn×2/一致性/ticker池/sector/fitness门槛）**没有一条限制持仓数**
- LLM 在"集中优先"+"fitness≥7 必须出现"等规则推动下，一路买到 7-8 只，prompt 的"3-5 只"形同虚设
- 不加这个约束，换手率怎么调都是治标——满仓后换仓永远会撞换手上限

**为什么换手率要改单向算法（再治标）**：
- 原 `sum(|delta|)` 是双向累加：卖 30% + 买 20% = 50%
- 这意味着满仓后**任何 2 只以上的换仓都必然超 40%**，数学上不可满足
- 改成 `max(总买入, 总卖出)`：同样换仓 = max(20%, 30%) = 30%，合理反映"真实换了多少仓位"
- 净加仓/净降仓场景（只买不卖/只卖不买）单向=双向，不受影响

**为什么不只改一个**：持仓数约束治本（防止再失控），换手率算法治标（让正常换仓不被误杀）。两者解决不同层面的问题——持仓数防"涨上去"，换手率防"换不动"。单做任何一个都不够：只限持仓数，正常换仓仍可能撞旧的双向换手上限；只改换手率，持仓还会再失控到 10 只。

**revise 反馈为什么也要改**：之前反馈只说"换手 0.549 超 0.4"，LLM 不知道该砍哪个动作。现在针对持仓超限/换手超限/行业超限分别给可执行指引（"砍 fitness 最低的""减少同时买卖对数""同行业别再 BUY"）。配合 max_revise_retries 2→3，给 LLM 更多收敛机会。

**最终方案（三层）**：
1. **持仓数硬约束**（治本）：规则 12，`target>0 的 action 数 ≤ max_positions(5)`，超了 revise
2. **换手率单向算法**（治标）：`max(总买入, 总卖出)`，参数 0.40→0.50
3. **revise 反馈 + retry**（放大器）：可执行指引 + 2→3 次

### 决策 G：撞锁死磕怎么修——为什么不动 7 天，而是加止盈豁免 + 变通反馈

**用户原问**："7 天是合理的吗？还是 3 天或者 5 天什么的"

**先回答"7 天合不合理"**（用数据，不拍脑袋）：
- `anti_churn_days: 7` 用**自然日**算（holdings-loader.ts:35），跨周末白占，实际锁 ≈5 个交易日
- **豫光金铅 case 证明锁的价值**：06-22 建仓，06-23（持仓 1 天）LLM 想卖被锁，到期末 +7.6%。锁住了，反而对了——如果第 2 天卖了就错过这 7.6%
- **结论：7 天不该改**。它的防 churn 价值被数据证实。shorten 到 3 天可能重新出现 churn（赛腾股份问题）

**真正的缺陷是什么**（两层，都不是天数问题）：

①**锁一刀切，不区分止损/止盈**：
- 之前只有"止损豁免"（破位/deal_breaker/建仓大回撤可突破锁）
- 但"止盈"（浮盈够多想落袋为安）一律被挡——豫光 +7.6% 想止盈换股，被锁挡死
- 止盈是合理操作（趋势涨到位了该走），不该被 churn 锁挡住
- **修：加 take_profit_threshold=0.15**，浮盈≥15% 时 takeProfitSignal=true，允许突破锁卖出。与止损镜像：锁防噪音，不锁真正的退出动机（止损下行 + 止盈上行）

②**LLM 撞锁后不会变通**：
- 06-23 LLM 想卖豫光金铅，撞锁，revise 3 次还是卖豫光金铅
- 它不知道"那改 HOLD 等解锁""改卖别的没锁的""只买不卖"
- **修：composeReviseFeedback 加 anti-churn 指引**——撞锁时明确告诉 LLM 三条出路

**为什么不只改一个**：
- 只加止盈豁免：撞锁死磕还在（LLM 还是不知道变通，只是浮盈够时能卖）
- 只加变通反馈：该止盈时还是被挡（LLM 学会变通了，但浮盈 +20% 想落袋也被锁）
- 两者解决不同子问题：止盈豁免修"该卖时被挡"，变通反馈修"撞了不知道绕开"

**阈值为什么是 15%**：太低（如 5%）→ 锁形同虚设，正常波动就触发，重新 churn；太高（如 30%）→ 锁死该止盈的。15% 是"明显趋势利润"的门槛，低于此视为"还没到止盈点，可能是 churn"。⚠️ 经验值，待更多回测数据校准。

**最终方案（两层，不动 7 天）**：
1. **止盈豁免**（修该卖被挡）：`take_profit_threshold: 0.15` + validator 规则 6 放行 takeProfitSignal
2. **撞锁变通反馈**（修撞了不会绕开）：composeReviseFeedback anti-churn 指引（改 HOLD/卖别的/只买不卖）

### 决策 H：行业偏科卡死——为什么不改 scan 也不放宽 25%，而是给 LLM 转向清单

**问题**：阶段 12 后最后两天卡死在"电子 37% 超 25% 上限"。深挖发现候选池 top_picks 电子占 60%+。

**为什么不改 scan 加行业分散约束**：
- scan 是趋势扫描，作用是选出最强主线——半导体周期是当时最强主线，选出大量电子股是**正确反映市场结构**
- 给 scan 加"每行业最多 N 只"约束，会把 fit9 的真强势电子股挡在门外，违背趋势策略"选最强"的原则
- 偏科在 rebalancer 这层处理更合适（它有 fitness + 约束的全局视角），不该让 scan 提前裁剪

**为什么不放宽 single_sector 25%→30%**：
- 治标不治本——下次市场更集中（如某行业占 70%+）时还会撞 30% 上限
- 弱化分散保护——单行业 25% 是合理的风险约束（行业集中度风险）
- 真正问题不是"25% 太严"，而是 LLM 撞了不会转向（候选池明明有非电子强标的）

**为什么改 revise 反馈是对的**：
- 数据证明候选池**有非电子强标的**（中科曙光 fit8、巨化 fit8、中国巨石 fit7）——LLM 不是无米之炊，只是没意识到该转向
- 给它具体的非超限行业强标的清单，它有明确出路，不会死磕电子
- 最小改动，不破坏趋势策略"选最强"的核心逻辑

**最终方案**：composeReviseFeedback 加 FeedbackContext：
- overSectors：从违规 plan 重算的超限行业集合（比解析 detail 字符串稳健）
- reports：候选股报告，筛 fit≥7 + sector 不超限 + 无 deal_breaker 的，作为"可转向配置"清单
- 反馈示例："电子 已满，不要再往这些行业 BUY/ADD。应转向【其他行业的强标的】。候选池中非电子行业的强标的（可转向配置）：中科曙光(计算机设备,fit8)、巨化股份(化工,fit8)、中国巨石(玻璃玻纤,fit7)"

**边界**：候选池真没有非超限强标的时（极端偏科），只给方向指引不列空清单——诚实告诉 LLM "确实没好选项"，而不是误导。

**深挖后扩展为三层**（验证发现单层转向反馈不够）：
- 旧 scan 重跑验证时，转向反馈给了"可转向配置：巨化股份"的建议，但巨化根本没进 top_n=10 的候选池——LLM 看不到它
- **补充第 1 层：rank 去「板块共振加分」**。这是偏科的源头——LONG/SHORT prompt 鼓励"同板块共振加分"，导致电子股互相推高扎堆。删掉后非电子强标的自然上来（巨化 top12→top3）
- **补充第 2 层：top_n 10→15**。让上来的非电子强标的进得了候选池（5 持仓 × 3 = 15 候选，给跨行业选股留空间）
- 最终方案：**去共振（源头）+ 扩 top_n（空间）+ 转向反馈（兜底）**，三层配合

### 决策 I：为什么默认模型换 glm-5.2 + evaluations/actions 为什么要语义分离

**问题**：06-26 空仓重跑 rebalance，glm-5-turbo 3 次撞 10 分钟超时。

**为什么不是简单调大超时**：trace 分析显示 turbo 单次调用 8-226s（巨化股份一次 226s），15 只串行（并发=2 实际接近串行）总耗时 15 分钟。这是**模型推理慢**，不是偶发卡顿——调大超时只是把痛苦延后，每次回测都要等 15 分钟。

**为什么 A/B 对比后选 glm-5.2**：
- **快 2.6×**（5.6min vs 15min）：前台命令不再超时，回测效率翻倍
- **fitness 更准**（avg 6.1 vs 5.4）：turbo 系统性压分（火炬电子 MLCC 龙头 +147% 趋势只给 4 分，5.2 给 7 分）
- **revise 更少**（0 vs 1 次）：对约束理解更准
- **成本持平**（$0.034）：输出 token +12% 但单价更低
- **无长尾**：turbo 的 226s 极端值在 5.2 降到 41s，并发效率高

**为什么不直接用 glm-4-flash（更快）**：flash 是非推理模型，fitness 质量未经回测验证。5.2 已用 trace 级数据证明质量优于 turbo，是"够快 + 更好"的甜点位。

**选股 0 重合不是换模型的阻碍**：深挖发现主因是"6+ 候选选 3 只"的数学概率（正常），叠加 turbo 的约束污染（异常）。换 5.2 后两者都改善。

**evaluations/actions 语义分离为什么是 prompt 层修复（方案 A），不是 post-processing（方案 B）**：
- 方案 A（prompt 分离）：让 LLM 从源头就不混淆"真实评估"和"约束执行"。改两处：输出格式段说明 + revise 反馈强调。一次修复，所有归档理由都干净
- 方案 B（后处理识别约束词）：只能标记不能消除，且要维护"约束词词典"，新措辞会漏网
- 选 A 的额外好处：即使换回 turbo 或其他模型，revise 时也不会污染——防御性加固

**稳定性验证为什么重要**：换模型后必须确认"同样的输入产出稳定的决策"。06-26 跑两遍的结论：
- 选股核心标的 100% 一致（国泰/润禾/晶瑞三只两次都买）
- fitness 波动在 ±1-2（核心买入标的稳定在 8，平均分 6.1→6.2 不变）
- 耗时波动 <6%
- 这说明 glm-5.2 的推理**方向稳定**，fitness 的轻微波动是 LLM 固有随机性（temperature=0 也不能完全消除），不影响决策

**边界**：fitness 的 ±1 波动对仓位计算有影响（fit8→9 仓位从 10.6%→11.9%），但对"买不买"的决策无影响（fit≥7 都会买）。波动集中在非核心标的，核心标的稳定。

## 九、如何重跑回测（操作备忘）

```bash
# 增量跑下一天（每天一个进程，跑了存盘）
node dist/backtest-cli.js --api-key <KEY>

# 从头重跑（改了参数后验证）
node dist/backtest-cli.js --reset
node dist/backtest-cli.js --api-key <KEY> --capital 200000

# 批量跑多天（慢路径，一次跑完，但超时不存盘）
node dist/backtest-cli.js --api-key <KEY> --dates 2026-06-17,2026-06-18,...

# 查看进度
node dist/backtest-cli.js --show

# 查看某天的调仓决策（评估表 + 动作理由 + 组合总结 + 约束博弈）
node dist/backtest-cli.js --day 2026-06-23

# 生成完整报告（含评分校准段）
node dist/backtest-cli.js --report
```

**注意**：
- LLM 调用是主要耗时（每只 20-60s），数据脚本有 4h 缓存
- snapshot 阶段慢（5207 股 × 查异动，并发 6 约 10 分钟），用 `--concurrency 6` 加速
- STATE_VERSION 当前 v3，v2 state 可兼容加载（entry_fitness 可选），v1 拒载
