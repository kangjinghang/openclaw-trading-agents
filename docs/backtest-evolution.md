## 一、定位（一切决策的起点）

```
账户：20 万小账户（A 股）
策略：趋势跟随（买异动动量延续，不抄底不猜顶）
风险偏好：可接受大回撤（换在场率）
目标持仓：3-5 只集中（非分散）
核心矛盾：小账户 × 高价股 × 趋势要在场
```

这五个定位决定了后面所有参数。如果定位变了（比如账户变大到 100 万，或转向价值投资），大部分参数需要重调。

## 二、架构总览

### 数据流（scan → rank → rebalance → backtest）

```
雪球异动榜
   │ scan_universe.py（5207 股全量）
   ▼
snapshot.py（逐只查异动，~10min，并发 6）
   │
   ▼  raw/snapshot-<date>.json
diff-cli（窗口期异动变化）
   │
   ▼  derived/diff-<date>.json
candidates-cli（ST/退市/科创板过滤）
   │
   ▼  derived/candidates-<date>.json  ← 候选池（无价格）
rank-cli（LLM 精排 top-N）
   │
   ▼  scan/<date>/scan.json          ← top_picks（无价格）
rebalancer（shallow-analyzer 拉数据 + LLM 出 action）
   │   ├─ position-calculator：fitness×0.022×波动率 → 目标仓位
   │   ├─ constraint-validator：单仓/行业/现金/anti-churn 七规则
   │   └─ revise 循环：违反约束→LLM 重出（最多 2 次）
   ▼  rebalance/<date>/plan.json     ← 含 industry（申万二级）
backtest-simulator（连续持仓模拟）
   │   ├─ normalizeWeights：价格漂移重算权重
   │   ├─ applyPlan：BUY/SELL/ADD/REDUCE + 手数取整
   │   └─ recordNav：每日 NAV 快照
   ▼  backtest/state.json            ← 增量续跑状态（只管持仓，不含决策原因）
   ▼  backtest/days/<date>/rebalance.json  ← 每日完整调仓决策归档
       （evaluations + actions[reason] + summary + reports[fitness] + 约束博弈）
backtest-cli --report
   ▼  backtest/run-<ts>/report.md    ← 含评分校准段
backtest-cli --day <date>
   ▼  终端渲染归档决策（人类可读视图）
```

### 四层仓位计算

```
LLM 出方向（BUY/SELL/HOLD/...）
        │ 不出数字，只出判断
        ▼
① 单股目标仓位 = fitness × 0.022 × 波动率折扣，single_name 截断
        │ computePosition（position-calculator.ts）
        ▼
② 全局现金排队：BUY 按 fitness 降序累计，超 spendable 的降级 HOLD
        │ applyPositions（position-calculator.ts）
        ▼
③ 硬约束校验：7 条规则，违反→revise
        │ constraint-validator.ts
        ▼
④ 成交手数取整：真实股数取整到 100 股，不足 1 手跳过
        │ backtest-simulator.ts applyPlan
```

### 阶段 1：rebalance 定位之争（06-27 上午）

**问题**：异动股池本质是"涨幅异动筛选"，A 股以炒作为主，备选常被 deal_breaker 跳过。用户质疑 rebalance 的价值。

**关键讨论**：用户明确——"我想让它自主决策，尽量不介入，可接受大回撤，用小账户做策略"。

**决策**：定调为**趋势跟随自主策略**。rebalance 的价值不是"帮用户决策"，而是"自主完成 scan→rank→深度分析→仓位→约束的全链路"。这决定了后续所有设计服务于"自主 + 趋势"。

---

### 阶段 2：趋势模式重构（06-27，commit `1bc2d68`）

**问题**：原 rebalancer 偏价值投资（PE/估值否决），和趋势定位冲突。

**决策**：
- 驱动逻辑评分（fitness 0-10）替代估值否决
- 技术位止损（支撑/压力/MACD）替代"高估就卖"
- 线性仓位（fitness×系数）替代"低估重仓高估轻仓"
- risk=high 不再压缩仓位（靠止损退出，不靠仓位）

**参数初值**：baseWeight 系数 0.008，single_name 0.10，cash_reserve 0.05。

---

### 阶段 3：增量回测模块（06-27，commit `20fcd5f` + `116e3cb`）

**问题**：单次全量回测要 20 分钟，用户"以为卡住了"。需要看到每天进度。

**决策**：
- `PositionSimulator.serialize()`/`fromSerialized()`：持仓状态可序列化
- state.json 跨进程持久化（STATE_VERSION 版本控制）
- 增量模式：每次只跑下一天，跑完存盘，断了能续
- 三种入口：默认增量 / `--date` 单日 / `--dates` 慢路径全量

**教训**：长任务必须有进度可见性（`[data] N/total` + `[llm] N/total` 日志），否则用户体验灾难。

### 阶段 4：仓位系数初调（06-27，commit `6cc501c`）

**问题**：0.008 系数下 fit8 才 6.4%，对 20 万账户 5-8 只目标仓位太轻。

**决策**：用户选"适度集中"，系数 0.008→0.015，single_name 0.10→0.15。

### 阶段 5：三层止损系统（06-27，3 个 commit）

**问题**：回测发现 anti-churn 锁（持仓<7天禁卖）挡住了所有止损信号，导致 0 次止损触发。

**决策**：三层止损，逐层解决不同的"卖不出去"问题：

| 层 | 解决的问题 | commit |
|----|-----------|--------|
| ① stopLossSignal 豁 anti-churn 锁 | risk=high/高危flag 被 7 天锁挡住 | `68b9b9b` |
| ② 建仓回撤止损 | 国瓷 -8.3% 没触发（建仓3天内） | `6e95edc` |
| ③ 技术破位强制 REDUCE | 生益 risk=high 但 PM 出 HOLD | `29bed07` |

**关键设计**：`held` map 的 `stopLossSignal` 字段在 position-calculator 之前**预判**，因为 validator 检查 anti-churn 锁在 position-calculator 之前。预判逻辑：overall_risk=high OR severity=高 OR deal_breaker OR 建仓回撤超阈值。

**第三层的特殊性**：代码强制减半（不留给 PM 犹豫）。因为发现 LLM 在 risk=high + 技术破位时仍出 HOLD，靠 LLM 自主止损不可靠。

### 阶段 10：调仓决策归档（06-28，可审计性修复）

**问题**：回测只能看到"结果"（NAV 曲线 + actions 简表），看不到"过程"——某天为什么买/卖某只股、跳过哪些候选、约束博弈了几轮，全部丢失。原因：`runSingleDay` 拿到了完整的 `rebalancer_output`（含 evaluations / 每个 action 的 reason / summary / reports），却只取 actions 的方向和名字，把 reason / summary / evaluations 全扔了。详细决策只残留在 `os.tmpdir()` 的 LLM trace 里，Windows 重启即清空。

**决策**：见决策推理链 E。每天把完整 `rebalancer_output` + `portfolio_before` + 每股 fitness/risk/thesis 报告 + 约束 revise 过程，原样归档到 `backtest/days/<date>/rebalance.json`。加 `--day <DATE>` 命令渲染成人类可读视图。

**职责分离**：`state.json` 只管持仓模拟（精简，已是增量续跑的状态机）；调仓"为什么"归 `days/<date>/`（按日独立文件，可审计、可 diff、可随时翻看）。两者不耦合——某天的决策归档坏了不影响持仓模拟续跑。

**当前状态**：已落地。历史 7 天需 `--reset` 重跑才能补归档（归档只在跑当天生成，无法事后回填，因为 LLM 输出是即时的）。

## 四、关键决策的推理链

### 决策 A：手数取整为什么在成交层，不在候选池过滤？

**用户原问**："先解决高价股问题，能直接候选池过滤吗"

**诊断**：数据流是 `candidates.json（无价格）→ rank → scan.json（无价格）→ rebalancer（拉 K 线才有 last_close）`。`filterCommon`（ranker.ts）拿到的 CandidateEntry **没有 price 字段**。价格要到 rebalancer 阶段才出现。

**结论**：候选池过滤做不了——在没价格的地方过滤价格，逻辑上不成立。

**更深一层**：固定价格阈值（如>200 元过滤）很粗糙。"买不买得起一手"取决于仓位大小，而仓位取决于 fitness（动态的）。245 元香农满仓 15% 能买 1 手，但 fitness 给 5% 仓位就买不起——这个判断不该在候选池做，该在成交时做。

**最终方案**：成交层手数取整。真实股数 = shares_norm × realCapital，取整到 lotSize，不足 1 手跳过。自适应仓位大小，不依赖固定价格阈值。

---

### 决策 B：行业分类为什么选申万一级映射，不选概念板块？

**用户原问**："A是映射成一级分类，按一级分类限制是吗？C是类似于概念板块？"

**三个候选**：
- A：申万二级→一级映射表（TS 硬编码）
- B：数据源改申万一级（Python 多一次网络调用）
- C：自定义产业链（手工定义"电子材料"等）

**为什么选 A**：
1. A 是修 bug（当前规则 3 形同虚设），不是加功能。先修 bug 再谈策略。
2. 映射表是确定性的（akshare `sw_index_second_info()` 快照，131 个二级稳定），不引入主观判断。
3. 改动极小（1 个纯函数 + 1 行调用），B 要改数据源增加失败面，C 要维护主观映射表。
4. 做完 A 重跑回测，如果规则 3 开始触发 revise，就证明分散约束确实在起作用——那时再评估要不要上 C。

**两个易错陷阱**（回测才发现）：
- `军工电子Ⅱ` → **国防军工**（不是电子！名字带"电子"但归军工）
- `PCB` → 电子（非标三级标签，东财 datacenter 特有，不在申万二级表）
- 东财 BOARD_NAME 会省略Ⅱ后缀（`电子化学品` 而非 `电子化学品Ⅱ`）

**设计要点**：映射只在规则 3 聚合时做，LLM prompt 里仍展示原始二级细分（不丢信息）。映射函数幂等（`mapIndustryToL1(mapIndustryToL1(x)) === x`），可重复调用安全。

### 决策 C：方向B 为什么是 baseWeight 0.022？

**死循环诊断**：

```
fitness 偏低（avg 5.3, max 8）
   ↓ 原因：评分标准过严（"有具体数据支撑才给7分以上"门槛太高）
baseWeight 给小仓（fit8 才 12%）
   ↓ 0.015 系数
3-4 只集中也满不了仓（42% 封顶）
   ↓ 数学：3×fit8成长股×0.8波动率 = 3×9.6% = 28.8%
要么放弃集中（买更多只→又回到分散问题）
要么放弃满仓（接受低仓位→资金闲置）
```

**为什么不是只调参数（B-1）或只改评分（B-2）**：
- 只调参数：fit5 的弱驱动股也给 15%，可能买入质量不高的标的。
- 只改评分：影响面大（所有分析师输出都变），且 fitness 偏低可能是这批候选股的特殊情况。
- 两者都做（B-3）：参数让中等分能上场，评分让高分股更多。最平衡。

**0.022 的数学验证**：
- fit8 成长股（0.8 折扣）：8×0.022×0.8 = 14.1%
- fit7 大盘股（1.0 折扣）：7×0.022×1.0 = 15.4%
- 5 只 fit7：77%（接近满仓）
- single_name 0.22：让 fit10（10×0.022=22%）不被截断

### 决策 D：评分校准为什么是基础设施优先？

**核心洞察**：`fitness × 0.022` 的整个仓位逻辑，建立在一个**从未验证的假设**上——"LLM 给的分数和股票实际涨跌正相关"。如果这个假设不成立：
- 高分股不比低分股涨得好 → 重仓高分股没有 alpha
- 甚至倒挂（高分股跌更多）→ 越重仓越亏

这是比"参数调多少"更根本的问题。

**为什么现在只做记录、不做结论**：
- 校准需要"建仓时 fitness + 平仓后收益"的配对样本
- 统计显著性要求 30+ 笔交易
- 7 天回测只有 1-2 笔平仓，远不够
- 所以先铺基础设施（entry_fitness 记录 + 校准段渲染），等数据够了自动出结论

**校准段的三种结论**：
- ✓ 校准良好：高分档收益 > 低分档，单调递减 → 评分可信，仓位公式有底气
- ⚠ 校准失准：高分档 < 低分档（倒挂）→ 评分无预测力，需调 shallow-analyzer prompt
- △ 校准模糊：差异小 → 区分度不足，或样本不够

---

### 决策 E：调仓决策归档为什么按日独立文件，不塞进 state.json？

**用户原问**："我看你找了很久，是不是现在的代码设计不合理？"

**诊断（用户一针见血）**：调仓的 reason / evaluations / summary 只在跑回测那一刻由 LLM 生成，然后**用完即弃**。`runSingleDay` 拿到完整 `rebalancer_output` 却只取 actions 的方向和名字，其余全丢。要回溯某天为什么这么调，得去 `os.tmpdir()` 的 LLM trace 里"考古"——那是系统临时目录，重启即清空。回测因此变成黑盒：只告诉你赚了多少，不告诉你每天怎么决定的。

**为什么不塞进 state.json**：
- state.json 是持仓模拟的状态机（cash + positions + navHistory + trades），职责单一
- 调仓决策体量大（每股含 thesis/key_signals/risk_flags + evaluations + 约束 violations），塞进去会让 state 膨胀到难以人工检查
- state 每天整体覆盖写（增量续跑），决策若是其中一部分，版本/回滚会混乱
- 按日独立文件可单独打开、单独 diff 两天差异、坏了不影响持仓续跑

**为什么用 `days/<date>/` 而非 `run-<ts>/`**：
- `run-<ts>/` 是 `--report` 的产物（一次性报告，每次生成新目录）
- `days/<date>/` 与回测日一一对应，稳定路径，"想看 06-23 怎么调的"直接打开固定文件
- 目录形式（非单文件）便于后续扩展（如加每日 K 线快照、shallow-analyzer 原始报告）

**为什么无法事后回填**：LLM 输出是即时的（每次跑带随机性 + 最新数据），历史日的决策无法重现。所以归档**必须在跑当天生成**，历史日要补只能 `--reset` 重跑。这本身也是个信号：可审计性要从第一天就做，不能事后补。

**最终方案**：
- `archiveDayRebalance()`：`runSingleDay` 里 rebalance 跑完立即归档，非 ok 日也存（失败日更要看）
- `--day <DATE>`：渲染归档成人类可读视图（评估表 + 动作理由 + 组合总结 + 约束博弈）
- 测试锁定渲染契约（backtest-archive.test.ts，5 个用例）

## 九、如何重跑回测（操作备忘）

```bash
# 增量跑下一天（每天一个进程，跑了存盘）
node dist/backtest-cli.js --api-key <KEY>

# 从头重跑（改了参数后验证）
node dist/backtest-cli.js --reset
node dist/backtest-cli.js --api-key <KEY> --capital 200000

# 批量跑多天（慢路径，一次跑完，但超时不存盘）
node dist/backtest-cli.js --api-key <KEY> --dates 2026-06-17,2026-06-18,...

# 查看进度
node dist/backtest-cli.js --show

# 查看某天的调仓决策（评估表 + 动作理由 + 组合总结 + 约束博弈）
node dist/backtest-cli.js --day 2026-06-23

# 生成完整报告（含评分校准段）
node dist/backtest-cli.js --report
```

**注意**：
- LLM 调用是主要耗时（每只 20-60s），数据脚本有 4h 缓存
- snapshot 阶段慢（5207 股 × 查异动，并发 6 约 10 分钟），用 `--concurrency 6` 加速
- STATE_VERSION 当前 v3，v2 state 可兼容加载（entry_fitness 可选），v1 拒载
